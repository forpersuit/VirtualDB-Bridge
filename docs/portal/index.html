Security Architecture & Zero Trust

安全架构与零信任防御

Scenario: Developer Mistake 场景：开发人员失误

A developer forgets to add a WHERE tenant_id = ? filter in the application code, potentially leaking all tenants' data. 开发人员在编写代码时忘记在 SQL 中添加 WHERE tenant_id = ? 过滤条件，导致一个租户看到了所有公司的数据。

Our Defense

SQL Virtualization forces the prefix at the gateway level. The developer *physically cannot* query across tenants.

SQL 虚拟化在网关层强制执行前缀注入。开发人员在物理上写不出能跨库查询的代码。

Scenario: Direct DB Attack 场景：数据库直连攻击

An external hacker discovers your origin database IP and attempts a brute-force login to your MariaDB/MySQL instance. 外部黑客发现了你后台数据库的真实 IP，并尝试直接暴力破解你的 MySQL 或 MariaDB 数据库。

Our Defense

Cloudflare Zero Trust hides your origin. The DB only accepts connections from the Bridge's internal IP.

Cloudflare 零信任隐藏了真实源站。数据库仅接受网关内网 IP 的连接，拒绝任何公网访问。

Scenario: Gateway Hijack 场景：网关遭遇劫持

An attacker compromises the VirtualDB-Bridge server and attempts to issue queries to another tenant's prefixed tables. 黑客攻破了网关服务器，并尝试利用网关的权限直接发送 SQL 指令去查询其他租户的前缀表。

Our Defense

DB Wildcard Permissions. The DB engine rejects the request because the user is only allowed to touch tenantA_%.

数据库通配符权限。数据库引擎会直接拒绝请求，因为该账号被限制只能操作 tenantA_%。

Production Deployment in 3 Steps 三步开启生产环境部署

STEP 01

DB Level Restriction

底层数据库权限限制

Use wildcard patterns to ensure each DB user can only see their own tenant prefix.

使用通配符模式，确保数据库用户仅能操作属于其租户前缀的库。

GRANT ALL PRIVILEGES 
ON `tenantA\_%`.* 
TO 'bridge'@'%';
STEP 02

Zero Trust Identity

零信任身份认证

Configure Cloudflare Access and generate a Service Token for your application.

配置 Cloudflare Access 并为你的客户端生成专属的 Service Token。

  • Save Client ID & Secret
  • Get Audience Tag
STEP 03

Launch Gateway

启动网关服务

Update your .env file with the collected keys and start the Node.js server.

将获取的密钥填入 .env 文件并启动 Node.js 服务器。

PORT=3002
DB_PASS=*******
CF_AUDIENCE_TAG=xxxx
npm start
